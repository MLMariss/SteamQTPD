## Axis 2 — Refresh schedule (is the pipeline keeping up?)

Covered rows split by how each row's **own scraper** will treat it on the next pass — NOT against a flat target. The **7d (active)** lane is games updated within 90d (short cooldown); the **30d (dormant)** lane is everyone else (long cooldown, refreshed rarely *by design*). Track totals **include** their overdue members. **overdue** = already past its lane's cooldown = real backlog. **never** = no data yet = fill frontier. **empty** = correctly skipped (below the 10-review floor / null score), not pending work.

| Metric | Storage | 7d-track | 30d-track | overdue | empty | never |
|---|---|---:|---:|---:|---:|---:|
| Recent reviews | `recent.json` | 11,598 | 112,646 | 465 | 0 | 19 |
| Playtime raw (approx) | `playtime_raw/` | 7,678 | 71,107 | 50,420 † | 44,894 | 584 |
| Update events | `updates_raw/` | 7,519 | 71,149 | 4,657 | 44,894 | 701 |

**†  Playtime `overdue` is proxy-inflated — read with caution.** `playtime_raw/` stores no per-game scrape timestamp, so staleness uses each game's **newest review `ts`** as a proxy. A game with no recent reviews reads as overdue even if the scraper walked it days ago, so this figure is an **upper bound**, not true backlog (contrast Update events, which has a real per-game `scraped_at` and shows exact overdue). The `empty` column is the 44,894 games below the 10-review floor (correctly skipped, not backlog). An exact figure needs a per-game `scraped_at` in the shards — see Future work.

Files with a single-window (not two-track) refresh rule:

- **`games.json` core** (`scraper.py`): fresh 24,246 · overdue 4 · never 0 · last-modified-only 100,013. Games within 365d of release are on the age-tiered review refresh (≤3d→0.25d, ≤10d→0.5d, ≤30d→1d, ≤60d→2d, ≤90d→3.5d, ≤180d→7d, ≤365d→15d), so fresh/overdue are exact for them. Older games refresh on Steam's `last_modified` (invisible locally) plus the PICS review-drift trigger, and are counted separately rather than scored against a cooldown they don't have.
- **`hltb.json`** (`hltb_refresh.py`): fresh 33,416 · overdue 52 · blank-frozen 0 · blank-active 90,776. Windows: partial 14d, full 365d; blanks back off 3→30→180d by attempt count.
- **`prices.json`** (`price_and_sale.py`): no cooldown gate — the whole non-free base is re-batched every 3h, so there is no meaningful staleness backlog to bucket.
- **`tags.json`** (`tags_refresh.py`): **no timestamp stored and no rescrape schedule** — coverage-only. See Future work below.
- **`playtime.json` / `ratings.json`**: derived from `playtime_raw/` on every raw pass; staleness inherits from the raw shard row above.
- **`pics_raw/`** (`pics_refresh.py`): fresh 124,230 · overdue 0 · never 33. Single flat `--stale-days 14` window (daily cron), and — unlike playtime — keyed on a **real per-game `_ts` fetch stamp**, so overdue here is exact, not a proxy. `never` = catalog games not yet in `pics_raw/` (the fill frontier). `pics/` is derived from `pics_raw/` by `pics_summarize.py` on every pass, so its staleness inherits from this row.

## Notes

**Prices.** `prices.json` holds **106,078 rows (85.4% of catalog)**, tracking the non-free base (**106,098** games after removing 18,165 `is_free`). **12,870 of 106,078 priced titles (12.1%) are currently discounted**; sale end-dates come from the `IStoreBrowseService/GetItems` pass.

**Playtime backfill.** Raw playtime covers **78,966 games (63.5% of catalog)**. The honest denominator is the addressable set after the `MIN_REVIEWS_FLOOR = 10` gate — **79,369 games (63.9% of catalog)** — against which raw is **99.5% of addressable**. The other **44,894 games are below the floor and correctly skipped** (too few reviews to produce a sentiment-split median).

**Summarizers run on every raw pass.** Raw 78,966 · summarized 78,802 · rated 78,797 (current gap 164). `playtime_summarize.py` and `ratings_summarize.py` are chained into `playtime-raw.yml`, so the small frontend files refresh right after the shards do (8×/day). Any residual gap is just within-run ordering between the shard commit and the summarize steps, not a stalled pipeline.

**Update events.** `updates.json` (event_type-based patch history) covers **51,087 games (41.1%)**, built from **64 of 64 `updates_raw/` shards populated**. This layer feeds the frontend's **Updated column cadence badge** (`N · 90d` / `N · 1y`, from the summed 90d/365d `counts`) and backfills a null News-API `last_update_ts`. It is gated by the same 10-review floor as playtime. As the remaining shards populate this coverage rises; the precedence flip (event layer becomes primary over News-API) is gated on that coverage — see ARCHITECTURE §9.5 / §3.1.

**HLTB estimates refresh live.** **18,824 of 124,244 `hltb.json` entries carry the `est` flag** (15.1%), leaving **105,420 reals (84.8%)**. `hltb_estimate.py` is a shared helper imported by `hltb_refresh.py` (as `HE`) and called in the fill loop, so estimates recompute from live median ratios every 2h as new reals land — not a stale one-off.

**Rating % / review count.** Rating % is present for **113,259 games (91.1%)**; review count for **116,602 (93.8%)**. The gap is titles with too few reviews to carry a meaningful score.

**Tags** are non-empty for **124,229 games (100.0%)**. The remainder are untagged on SteamSpy (typically very-low-review or unreleased titles).

## Future work

- **Tag refresh schedule.** `tags.json` currently has no per-entry timestamp and no rescrape cadence — tags are fetched once and left. Tags do drift (Steam user-tag votes shift, new tags get added), so a light periodic re-check (e.g. weekly/monthly, oldest-first, small budget) is worth adding. That needs a per-entry `scraped_at` in `tags.json` first, after which tags can join Axis 2 as a proper row here.
- **Uniform-refresh review.** Once the fill frontier drains (near-100% coverage), revisit whether the dormant 30–45d cooldowns should tighten toward a flatter, faster whole-catalog refresh. Today's split (active 4–7d / dormant 30–45d) deliberately front-loads budget onto games whose data actually moves; a uniform target only makes sense once no backlog is competing for that budget.
- **Playtime per-game timestamp.** Adding a per-game `scraped_at` to the `playtime_raw/` shard records would replace the newest-review-`ts` proxy with an exact staleness signal, removing the (approx) caveat above.
