## Axis 2 — Refresh schedule (is the pipeline keeping up?)

Covered rows split by how each row's **own scraper** will treat it on the next pass — NOT against a flat target. The **7d (active)** lane is games updated within 90d (short cooldown); the **30d (dormant)** lane is everyone else (long cooldown, refreshed rarely *by design*). Track totals **include** their overdue members. **overdue** = already past its lane's cooldown = real backlog. **never** = no data yet = fill frontier. **empty** = correctly skipped (below the 10-review floor / null score), not pending work.

**†  Playtime `overdue` is proxy-inflated — read with caution.** `playtime_raw/` stores no per-game scrape timestamp, so staleness uses each game's **newest review `ts`** as a proxy. A game with no recent reviews reads as overdue even if the scraper walked it days ago, so this figure is an **upper bound**, not true backlog (contrast Update events, which has a real per-game `scraped_at` and shows exact overdue). The `empty` column is the 44,832 games below the 10-review floor (correctly skipped, not backlog). An exact figure needs a per-game `scraped_at` in the shards — see Future work.

## Notes

**Prices.** `prices.json` holds **105,424 rows (85.3% of catalog)**, tracking the non-free base (**105,445** games after removing 18,129 `is_free`). **13,393 of 105,424 priced titles (12.7%) are currently discounted**; sale end-dates come from the `IStoreBrowseService/GetItems` pass.

**Playtime backfill.** Raw playtime covers **78,629 games (63.6% of catalog)**. The honest denominator is the addressable set after the `MIN_REVIEWS_FLOOR = 10` gate — **78,742 games (63.7% of catalog)** — against which raw is **99.9% of addressable**. The other **44,832 games are below the floor and correctly skipped** (too few reviews to produce a sentiment-split median).

**Summarizers run on every raw pass.** Raw 78,629 · summarized 78,457 · rated 78,451 (current gap 172). `playtime_summarize.py` and `ratings_summarize.py` are chained into `playtime-raw.yml`, so the small frontend files refresh right after the shards do (8×/day). Any residual gap is just within-run ordering between the shard commit and the summarize steps, not a stalled pipeline.

**Update events.** `updates.json` (event_type-based patch history) covers **30,218 games (24.5%)**, built from **38 of 38 `updates_raw/` shards populated**. This layer feeds the frontend's **Updated column cadence badge** (`N · 90d` / `N · 1y`, from the summed 90d/365d `counts`) and backfills a null News-API `last_update_ts`. It is gated by the same 10-review floor as playtime. As the remaining shards populate this coverage rises; the precedence flip (event layer becomes primary over News-API) is gated on that coverage — see ARCHITECTURE §9.5 / §3.1.

**HLTB estimates refresh live.** **18,762 of 123,558 `hltb.json` entries carry the `est` flag** (15.2%), leaving **104,796 reals (84.8%)**. `hltb_estimate.py` is a shared helper imported by `hltb_refresh.py` (as `HE`) and called in the fill loop, so estimates recompute from live median ratios every 2h as new reals land — not a stale one-off.

**Rating % / review count.** Rating % is present for **112,331 games (90.9%)**; review count for **115,642 (93.6%)**. The gap is titles with too few reviews to carry a meaningful score.

**Tags** are non-empty for **123,524 games (100.0%)**. The remainder are untagged on SteamSpy (typically very-low-review or unreleased titles).

## Future work

- **Tag refresh schedule.** `tags.json` currently has no per-entry timestamp and no rescrape cadence — tags are fetched once and left. Tags do drift (Steam user-tag votes shift, new tags get added), so a light periodic re-check (e.g. weekly/monthly, oldest-first, small budget) is worth adding. That needs a per-entry `scraped_at` in `tags.json` first, after which tags can join Axis 2 as a proper row here.
- **Uniform-refresh review.** Once the fill frontier drains (near-100% coverage), revisit whether the dormant 30–45d cooldowns should tighten toward a flatter, faster whole-catalog refresh. Today's split (active 4–7d / dormant 30–45d) deliberately front-loads budget onto games whose data actually moves; a uniform target only makes sense once no backlog is competing for that budget.
- **Playtime per-game timestamp.** Adding a per-game `scraped_at` to the `playtime_raw/` shard records would replace the newest-review-`ts` proxy with an exact staleness signal, removing the (approx) caveat above.
